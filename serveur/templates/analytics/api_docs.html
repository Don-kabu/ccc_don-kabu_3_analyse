Digital Scholar — Référence technique

Documentation de l'API

Intégrez le suivi analytique de votre application cliente avec le tableau de bord Digital Scholar. Deux endpoints REST pour envoyer vos événements en temps réel.

BASE URL http://<votre-serveur>

2 endpoints REST

Un pour les événements de lecture, un pour les actions utilisateur.

Sans authentification

Les endpoints de collecte sont ouverts — à sécuriser par IP ou token selon vos besoins.

Format JSON

Toutes les requêtes et réponses sont au format application/json.

Référence

Endpoints disponibles

POST /api/analytics/ Événement de lecture

Envoie un événement de suivi de section — temps passé par le lecteur sur une partie précise d'un article (intro, contenu, conclusion…). Ces données alimentent les métriques de rétention et d'analyse structurelle dans le tableau de bord.

Corps de la requête (JSON)

Champ Type Requis Description
article_id integer requis Identifiant de l'article lu (blog_article.id).
section string requis Nom de la section : intro, objectives, content, conclusion, resources.
duration_seconds integer requis Durée en secondes passées sur cette section au cours de la session.
session_id string optionnel Identifiant de session anonyme pour dédupliquer les lectures (max 80 chars).

Exemple de requête

JSON
// POST /api/analytics/
// Content-Type: application/json

{
  "article_id": 42,
  "section": "content",
  "duration_seconds": 187,
  "session_id": "sess_a3f9bc12"
}

Réponses

200
{ "ok": true }
400
{ "ok": false, "error": "invalid json" }
POST /api/action-log/ Journal d'action

Envoie une entrée de journal traçant une action utilisateur (vue d'article, navigation, recherche…). Ces événements constituent la source principale des graphiques temporels, du classement des enseignants et de l'explorateur de mesures.

Corps de la requête (JSON)

Champ Type Requis Description
action string requis Type d'action : article_view, home_view, tag_filter, comment_post…
actor_type string optionnel anonymous (défaut) ou user.
user_id integer | null optionnel ID de l'utilisateur connecté. null pour les visiteurs anonymes.
article_id integer | null optionnel ID de l'article concerné par l'action.
request_path string optionnel Chemin de la requête HTTP, ex. /articles/mon-titre/.
request_method string optionnel Méthode HTTP : GET, POST…
ip_address string optionnel Adresse IP du client (max 64 chars).
user_agent string optionnel User-Agent du navigateur pour la détection d'appareil (max 255 chars).
session_id string optionnel Identifiant de session anonyme (max 80 chars).
payload object optionnel Données contextuelles supplémentaires au format JSON libre.

Exemple de requête

JSON
// POST /api/action-log/
// Content-Type: application/json

{
  "action": "article_view",
  "actor_type": "anonymous",
  "user_id": null,
  "article_id": 42,
  "request_path": "/articles/pourquoi-le-ciel/",
  "request_method": "GET",
  "ip_address": "203.0.113.57",
  "user_agent": "Mozilla/5.0 (iPhone; CPU iPhone OS 17_0)",
  "session_id": "sess_a3f9bc12",
  "payload": { "referrer": "https://google.com" }
}

Réponses

200
{ "ok": true }
400
{ "ok": false, "error": "invalid json" }

Sécurité

Authentification

Endpoints de collecte ouverts : les routes /api/analytics/ et /api/action-log/ n'exigent pas de token pour permettre l'envoi depuis le client sans friction. En production, il est conseillé de les restreindre à votre IP cliente ou d'ajouter un en-tête X-Api-Key côté middleware.
Tableau de bord protégé : l'accès aux pages d'analyse (/dashboard/, /explorateur/, etc.) requiert une connexion via le formulaire d'authentification.

Référence

Codes de réponse HTTP

Code Signification Causes fréquentes
200 OK Succès La requête a été reçue et traitée.
400 Bad Request Corps invalide Payload non-JSON, encoding incorrect, corps vide.
405 Method Not Allowed Méthode incorrecte Utilisation de GET au lieu de POST sur un endpoint de collecte.
302 Found Redirection Accès à une page protégée sans être authentifié — redirige vers /accounts/login/.

Guide

Intégration depuis Django (client)

Ajoutez ces constantes dans settings.py de votre application cliente, puis appelez les endpoints depuis vos signaux ou middlewares de tracking.

settings.py
# Client_blog_post/Client_blog_post/settings.py

ANALYTICS_SERVER_URL = "http://127.0.0.1:8001"
ANALYTICS_ENDPOINT   = ANALYTICS_SERVER_URL + "/api/analytics/"
ACTION_LOG_ENDPOINT  = ANALYTICS_SERVER_URL + "/api/action-log/"
Python — envoi
import requests
from django.conf import settings

def push_action_log(action, article_id=None, request=None):
    payload = {
        "action": action,
        "actor_type": "anonymous",
        "article_id": article_id,
        "request_path": request.path if request else "",
        "user_agent": request.META.get("HTTP_USER_AGENT", "") if request else "",
    }
    try:
        requests.post(settings.ACTION_LOG_ENDPOINT, json=payload, timeout=2)
    except requests.exceptions.RequestException:
        pass  # ne jamais bloquer la réponse principale
