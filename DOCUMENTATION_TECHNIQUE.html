📌 1. Introduction

1.1 Objectif du document

Ce document décrit l'architecture, le fonctionnement technique et les interfaces du projet EduTrack CMS. Il est destiné aux développeurs, architectes logiciels, testeurs QA et responsables DevOps impliqués dans le développement ou la maintenance du système.

1.2 Portée du projet

EduTrack CMS est un système de gestion de contenu pédagogique destiné à la publication d'articles éducatifs pour enfants. Le système est composé de deux applications Django indépendantes :

  • Client (Client_blog_post) — Application publique de publication, lecture, commentaire et partage d'articles pédagogiques, avec un dashboard auteur intégrant des graphiques analytiques.
  • Serveur (serveur / Digital Scholar) — Serveur d'analyse centralisé qui consomme les logs d'actions du client via une connexion SQLite read-only directe, et offre un tableau de bord analytique avancé.

Fonctionnalités principales :

  • Création/modification/suppression d'articles pédagogiques (éditeur rich-text Quill)
  • Système de catégories (20 catégories éducatives) et de tags
  • Commentaires imbriqués (threaded) avec modération
  • Partage d'articles avec compteur de partages
  • Recherche plein texte pondérée (scoring multi-champs)
  • Suivi analytique complet : lectures, temps de lecture par section, OS, navigateur, appareil
  • Dashboard auteur avec graphiques Chart.js (barres, lignes, doughnuts)
  • Serveur d'analyse avec dashboard, explorateur, tendances
  • Système de logs d'actions (ActionLog) avec livraison vers le serveur

🏗️ 2. Vue d'ensemble du système

2.1 Description générale

Contexte métier : Les enseignants et créateurs de contenu éducatif ont besoin d'une plateforme simple, inspirée de Substack, pour publier des articles structurés (introduction, objectifs, contenu, conclusion, questions de réflexion, ressources) et suivre l'engagement des lecteurs.

Problème résolu : EduTrack centralise la création, la diffusion et l'analyse de contenus pédagogiques en un seul outil, éliminant le besoin de combiner un CMS générique avec des outils d'analytics externes.

2.2 Architecture globale

Le système suit une architecture bi-applicative monolithique : deux applications Django autonomes communiquant via une base de données SQLite partagée (lecture seule côté serveur) et des webhooks HTTP (push de logs).

3.2 Description des modules

Module Authentification

  • Inscription auteur : Formulaire personnalisé (AuthorSignupForm) avec validation email unique, champs prénom/nom obligatoires.
  • Connexion/Déconnexion : Django auth intégré (django.contrib.auth), templates personnalisés, redirection vers le dashboard après login.
  • Contrôle d'accès : Décorateur @login_required sur les vues auteur (dashboard, création, modification, suppression).
  • Lecteurs anonymes : Pas de compte requis pour lire, commenter ou partager.

Module Gestion des articles

  • Création : Éditeur Substack-style avec Quill.js pour intro/contenu/conclusion. Seul le titre est requis.
  • Catégorisation : 20 catégories éducatives prédéfinies + tags libres (many-to-many).
  • Médias : Upload ou URL pour les illustrations. Sanitisation HTML via bleach.
  • Workflow : Brouillon → Publié. published_at automatiquement défini à la première publication.
  • Recherche : Scoring pondéré multi-champs (titre×12, tags×10, intro×8, contenu×6, etc.).

Module Commentaires

  • Commentaires threadés : Modèle auto-référencé (parent FK vers self).
  • Formulaire inline : Nom requis, email optionnel, message (max 1000 caractères avec compteur).
  • Réponses : Formulaires de réponse inline avec annulation.
  • Modération : Champ is_approved (défaut True, filtré côté requête).

Module Analytique (Client)

  • Tracking de section : analytics.js envoie le temps passé par section (intro, objectifs, contenu, conclusion, ressources).
  • ActionLog : Enregistre chaque action (view, comment, share, login, etc.) avec IP, user-agent, session.
  • Livraison : Push HTTP vers le serveur d'analyse avec statut de livraison (pending/sent/failed).
  • Dashboard auteur : Articles publiés/jour, commentaires/jour, visites/jour, OS/navigateur/appareil, résumé commentaires.

Module Partage

  • Modal de partage : Popup avec copie du lien dans le presse-papier.
  • Compteur de partages : Chaque copie de lien crée un ActionLog(action='article_share') via endpoint AJAX dédié.
  • Affichage : Badge compteur sur le bouton « Partager », pills dans les listes et la page d'accueil.

Module Analytique (Serveur — Digital Scholar)

  • Dashboard : Lectures mensuelles, articles top, enseignants les plus actifs.
  • Explorateur : Filtrage par date, enseignant, catégorie, type d'appareil.
  • Tendances : Analyse temporelle des lectures et de l'engagement.
  • Accès DB client : Lecture directe via raw SQL (db_utils.py) en mode read-only.

4.4 Authentification & sécurité

  • Session-based auth : Django sessions avec cookies HTTPONLY.
  • CSRF : Activé globalement. Exceptions uniquement pour les endpoints d'API machine-to-machine (@csrf_exempt).
  • Gestion des rôles : Deux profils — Auteur (CRUD articles + dashboard) et Lecteur anonyme (lecture + commentaire + partage).

🚀 8. Performance & scalabilité

Optimisation des requêtes

  • select_related('author') et prefetch_related('tags') sur toutes les listes d'articles pour éviter les requêtes N+1.
  • Annotations en base (Count, annotate) pour view_count, comment_count, share_count plutôt que calcul Python.
  • Recherche par scoring SQL (Case/When) pour éviter des recherches en mémoire.
  • db_index=True sur category pour filtrage rapide.

Limites de taille

  • DATA_UPLOAD_MAX_MEMORY_SIZE : 20 MB (corps de requête)
  • FILE_UPLOAD_MAX_MEMORY_SIZE : 10 MB (fichiers uploadés)

Scalabilité

  • Actuel : SQLite mono-fichier, adapté pour un déploiement mono-serveur (jusqu'à ~100 utilisateurs simultanés).
  • Évolution recommandée : Migration vers PostgreSQL pour la concurrence d'écriture, ajout de Redis pour le cache de sessions et les compteurs temps réel.
  • CDN : Chart.js et Quill sont déjà chargés via CDN, réduisant la charge statique serveur.

Points d'attention

  • SECRET_KEY hardcodée en dev — DOIT être remplacée par une variable d'environnement en production.
  • DEBUG = True en dev — DOIT être False en production.
  • ALLOWED_HOSTS = [] — DOIT être configuré en production.
  • Les endpoints @csrf_exempt (track_analytics, track_share) acceptent des requêtes non authentifiées — rate limiting recommandé en production.

🛠️ 10. Maintenance & monitoring

Logs applicatifs

  • ActionLog : Table centralisée de tous les événements utilisateur avec statut de livraison, tentatives, erreurs.
  • Django logging : Configuration standard via settings.LOGGING (recommandé en production).
  • Livraison : Chaque log a un delivery_status (pending/sent/failed) + last_delivery_error pour diagnostic.

Monitoring recommandé

  • Métriques : Prometheus + Django-prometheus pour métriques HTTP (latence, taux d'erreur).
  • Dashboards : Grafana pour visualiser les métriques serveur.
  • Alertes : Alerte sur taux d'erreur > 5%, temps de réponse > 2s, espace disque SQLite.
  • Backup DB : Cron quotidien de copie des fichiers db.sqlite3.

Administration

  • Interface admin Django accessible sur /admin/ pour les deux applications.
  • Modèles enregistrés : Tag, Article, ReaderComment, ArticleAnalytics, AnalyticsEvent, ActionLog.
  • Commande de seed : python manage.py seed_demo_data pour données de test.

12.3 Architecture & technique

  • Migration PostgreSQL : Remplacer SQLite par PostgreSQL pour supporter la concurrence d'écriture, les recherches full-text natives (SearchVector, SearchRank) et les index GIN/GiST.
  • Cache Redis : Mise en cache des pages d'accueil et des compteurs (vues, partages) avec invalidation ciblée. Réduirait significativement la charge DB.
  • API REST complète : Exposer les endpoints via Django REST Framework pour permettre des clients mobiles ou SPA (React/Vue).
  • Tâches asynchrones (Celery) : Déporter l'envoi des webhooks analytics et les emails de notification dans des workers asynchrones.
  • Recherche Elasticsearch : Pour une recherche full-text avancée avec suggestions, auto-complétion, surbrillance des résultats et filtres à facettes.
  • Stockage cloud (S3/MinIO) : Remplacer le stockage local media/ par un bucket S3 pour la scalabilité et la résilience des uploads.
  • Conteneurisation Docker : Dockerfile + docker-compose pour standardiser l'environnement de développement et simplifier le déploiement.

12.4 Sécurité & qualité

  • Rate limiting : Limiter les endpoints publics (track_analytics, track_share, commentaires) avec django-ratelimit pour prévenir les abus.
  • CAPTCHA : Ajouter un CAPTCHA (hCaptcha ou Turnstile) sur les formulaires de commentaire et d'inscription pour bloquer le spam.
  • CSP (Content Security Policy) : Ajouter les en-têtes CSP pour restreindre les sources de scripts/styles et atténuer les risques XSS résiduels.
  • Audit de sécurité automatisé : Intégrer pip-audit et bandit dans le pipeline CI pour détecter les dépendances vulnérables et les anti-patterns.
  • Tests automatisés : Écrire une suite complète de tests (unitaires + intégration) pour atteindre la couverture cible de 85%.
  • HTTPS obligatoire : Forcer HTTPS en production avec SECURE_SSL_REDIRECT, SECURE_HSTS_SECONDS et cookies sécurisés.

12.5 Fonctionnalités avancées (long terme)

  • Système de rôles multi-niveaux : Éditeur, Relecteur, Auteur, Admin — avec workflow de validation (soumission → relecture → publication).
  • Versioning des articles : Historique des modifications avec diff visuel et possibilité de revenir à une version antérieure.
  • Contenu multilingue : Support i18n pour publier des articles en plusieurs langues avec django-modeltranslation.
  • Système de newsletter : Permettre aux lecteurs de s'abonner par email pour recevoir les nouveaux articles.
  • Gamification : Badges et niveaux pour les auteurs (ex: 10 articles publiés, 100 commentaires reçus) et les lecteurs (commentaires, partages).
  • Export PDF : Générer un PDF formaté de l'article pour impression ou partage hors-ligne (via weasyprint ou xhtml2pdf).
  • Intégration LMS : Connecter EduTrack à des plateformes LMS (Moodle, Canvas) via LTI pour intégrer les articles dans des cours.
